# v0.3 Implementation Plan

Date: 2026-06-01
Status: draft

Objective: add a minimal local wide-search runtime that turns the current Codex skill package into an executable, evidence-backed research workflow.

## Phase 0: Preserve Current Skill Quality

Tasks:

- Keep `README.md` user-first.
- Keep `wide-search-mode.md` readable and non-provider-first.
- Keep `docs/HARNESS_INTEGRATION.md` as the advanced maintainer surface.
- Add CI for current packaging checks before runtime code grows.

Acceptance:

- temp `CODEX_HOME` install works
- YAML and shell syntax checks pass
- no local path leaks
- wide-search reference installs with the skill

## Phase 1: Runtime Skeleton

Status: started. The initial fixture-backed runtime creates run metadata, research plan, source ledger, claim ledger, synthesis, and verification report.

Proposed path:

```text
runtime/wide-search/
  package.json
  src/
    cli.mjs
    run-store.mjs
    planner.mjs
    source-ledger.mjs
    claim-ledger.mjs
    scorer.mjs
    synthesizer.mjs
    verifier.mjs
    providers/
      fixture-provider.mjs
      command-provider.mjs
  fixtures/
  tests/
```

Tasks:

- create Node ESM package
- add CLI command for `run`, `verify`, and `inspect`
- create run directory under `.runs/wide-search/<run-id>/`
- write `run.json` and `research-plan.json`

Acceptance:

- fixture run creates run directory
- inspect prints concise run summary
- tests run without network

## Phase 2: Source And Claim Ledgers

Status: started. The runtime now supports `fixture` and `local-command` profiles, creates source and claim ledgers, and verifies unsupported claims.

Tasks:

- implement `source-ledger.jsonl`
- implement source score model
- implement accepted/rejected decisions with reasons
- implement `claim-ledger.jsonl`
- support claim-to-source mapping

Acceptance:

- fixture provider creates accepted and rejected source rows
- unsupported claim fixture fails verification
- duplicate source fixture produces warning or failure based on threshold

## Phase 3: Synthesis And Readable Report

Tasks:

- generate `synthesis.md`
- render top findings, ranked shortlist, coverage, evidence paths, and next human check
- keep implementation details out of the readable report
- include execution profile in report metadata

Acceptance:

- synthesis is useful without opening JSON files
- evidence paths are present
- report clearly states if search execution was unavailable or partial

## Phase 4: Verification

Tasks:

- validate required files exist
- validate accepted source count against depth
- validate important claims have source ids
- validate freshness metadata for time-sensitive claims
- validate duplicate ratio
- produce `verification-report.json`

Acceptance:

- positive fixture passes
- negative fixtures fail for clear reasons
- warning cases do not block unless threshold is exceeded

## Phase 5: Skill Integration

Tasks:

- document how `$kimi-agent-swarm-prompt` discovers the runtime
- add `KIMI_SWARM_RUNTIME_DIR` or reuse `KIMI_SWARM_HARNESS_DIR`
- update `execution-modes.md` with the local runtime command
- add example prompt and expected output shape

Acceptance:

- skill can stop at approval card when runtime is unavailable
- skill can route to runtime when configured
- final answer reports readable synthesis and evidence paths

## Phase 6: CI And Release Gates

Tasks:

- add GitHub Actions workflow
- run packaging checks
- run runtime tests
- run fixture benchmark smoke
- publish release checklist update

Acceptance:

- CI passes on pull request
- benchmark smoke is deterministic
- docs describe v0.3 capability without overclaiming Kimi parity

## Recommended First PR

Scope:

- add CI
- add runtime skeleton
- add fixture provider
- add run directory creation
- add minimal tests

Do not include:

- live web providers
- hosted Kimi integration
- distributed workers
- dashboards

## Risks

- Runtime can become too complex if provider abstraction is designed before fixture behavior is stable.
- Search quality may look better in prose than in verifier output.
- Users may still expect hosted Kimi Agent Swarm parity; docs must keep execution profile visible.

## Definition Of Done For v0.3

- A user can run one fixture-backed wide-search task locally.
- The run produces research plan, source ledger, claim ledger, synthesis, and verification report.
- The verifier catches at least one unsupported-claim negative fixture.
- README and skill docs remain user-first.
- CI runs packaging checks and runtime tests.
