# 9.8+ Product Excellence Plan

> Turning `kimi-agent-swarm-skill` from a promising skill pack into a repeatable, trust-worthy, community-loved open-source product.

## 1. Executive Summary

**Current score:** ~7.2 / 10  
**Target score:** 9.8+ / 10  
**Target user:** Knowledge workers who need researched, evidence-backed answers and are willing to pay for API usage, but want the tool to be predictable, auditable, and easy to integrate into their workflow.

**The gap:** The project today is a well-built demo/POC with good CI and docs. To reach 9.8+, it must become a **product** that external users install, run repeatedly, and recommend without hand-holding. That requires five shifts:

1. From "skill pack" to **batteries-included CLI product**.
2. From "API key required" to **zero-config first run** with clear upgrade path.
3. From "read the docs" to **self-healing, self-explaining UX**.
4. From "trust the model" to **deterministic verification + reproducible benchmarks**.
5. From "maintainer-driven" to **community-driven** contribution and support.

## 2. North Star Metric

**Weekly Active Users running at least 3 wide-search runs (WAU3)**

Why: A user who runs 3+ times in a week has moved beyond curiosity into a real workflow.

Supporting metrics:
- 7-day retention after first run ≥ 40%
- NPS ≥ 50
- GitHub issues resolved within 7 days ≥ 80%
- CI flake rate < 0.5%
- Average time-to-first-result (TTFR) < 3 minutes

## 3. User Personas

### 3.1 The Solo Analyst (Primary)
- **Who:** Consultant, investor, founder, researcher.
- **Job-to-be-done:** "I need a landscape map of X with sources and caveats by tomorrow."
- **Pain:** Building this manually takes 4-8 hours; hiring a junior researcher is expensive and slow.
- **Willing to pay:** $5-50 per deep research run for the right answer.
- **Needs:** Fast setup, transparent cost, readable output, evidence files for audit.

### 3.2 The Engineering Team Lead
- **Who:** Leads a small team evaluating tools, repos, or vendors.
- **Job-to-be-done:** "We need a comparison matrix of 10 candidates with trade-offs."
- **Pain:** Vendor websites are biased; HN/Reddit is noisy; internal wiki rots.
- **Needs:** Reproducible runs, shareable evidence, integration with Notion/Slack/Confluence.

### 3.3 The Open Source Power User
- **Who:** Already uses Kimi Code CLI, Codex, or similar tools.
- **Job-to-be-done:** "I want a better wide-search skill than the default."
- **Pain:** Default tools are too chatty, not evidence-first, and hard to audit.
- **Needs:** Clear capability boundary, honest non-parity messaging, hackable provider system.

## 4. Product Requirements

### 4.1 Zero-Config First Run (P0)

A new user should get a useful result in < 3 minutes:

```bash
git clone https://github.com/min9lin9/kimi-agent-swarm-skill.git
cd kimi-agent-swarm-skill
./bin/kasw research "open-source AI browser agent repos" --depth standard
# → readable answer + evidence files in ./.runs/wide-search/...
```

Requirements:
- Single binary/script entry point (`bin/kasw`).
- Mock provider works without any API key.
- First-run wizard suggests providers and helps set API keys.
- Clear error messages when API keys are missing.

### 4.2 Transparent Cost & Budget Control (P0)

Users must never be surprised by cost.

- Pre-run cost estimate: "This will use ~10 provider calls and ~50K tokens. Estimated cost: $0.03-$0.08."
- Runtime budget enforcement: `--max-cost-usd 1.00`, `--max-provider-calls 50`, `--max-tokens 500000`.
- Usage report after every run: calls, tokens, estimated cost, provider name.
- Dry-run mode: `--dry-run` shows planned subagents/queries without executing.

### 4.3 Reproducibility (P0)

- Every run produces a `run.json` with deterministic inputs and seeds.
- `--replay <run-id>` re-runs with the same prompt contract.
- `--seed <number>` for deterministic provider sampling where supported.
- Lock file for provider versions and fixture versions.

### 4.4 Provider Ecosystem (P1)

Ship at least 5 stable providers out of the box:

1. `mock` — zero-cost demo/CI.
2. `serper` — Google Search (current).
3. `tavily` — AI-native search.
4. `brave` — privacy-focused search.
5. `github` — GitHub code/repo search.
6. `arxiv` — academic paper search (P2).

Each provider:
- Has its own env var key name.
- Reports per-call cost.
- Has unit tests with recorded fixtures.
- Documents rate limits and known failure modes.

### 4.5 Output Formats & Integrations (P1)

- Markdown synthesis (current).
- JSON export for downstream tools.
- CSV table export for comparison tasks.
- Notion page export.
- Slack/Discord webhook summary.
- Obsidian vault integration (P2).

### 4.6 Benchmark Suite as Product Proof (P0)

A public benchmark suite that proves quality:

- `fixture-asset-mgmt`
- `fixture-sellside-research`
- `fixture-youtube-niche`
- `fixture-paul-graham-corpus`
- `fixture-github-repo-landscape`
- `fixture-market-scan`

Each benchmark:
- Has a golden answer / rubric.
- Runs in CI.
- Scores precision, recall, and citation accuracy.
- Results published in `BENCHMARKS.md`.

### 4.7 Distributed Profile (P2)

For users who need scale beyond 128 subagents:

- In-process worker queue (v0.6).
- Redis-backed queue (v0.8).
- Resume and caching.
- Cost-aware scheduler.

## 5. Technical Architecture

### 5.1 Layers

```text
┌─────────────────────────────────────────────┐
│  CLI / API / Skill Interface                │
├─────────────────────────────────────────────┤
│  Workflow Engine (prompt contract, approval)│
├─────────────────────────────────────────────┤
│  Wide-Search Runtime                        │
│  - planner                                  │
│  - provider registry                        │
│  - scorer                                   │
│  - verifier                                 │
│  - synthesizer                              │
├─────────────────────────────────────────────┤
│  Providers (mock, serper, tavily, ...)      │
├─────────────────────────────────────────────┤
│  Evidence Store (.runs/wide-search/<run-id>)│
└─────────────────────────────────────────────┘
```

### 5.2 Key Design Decisions

- **Provider abstraction:** Every search backend implements `SearchProvider`. Users can add custom providers by pointing to a file.
- **Ledger-first:** All decisions are recorded in machine-readable ledgers before synthesis.
- **Verification as gate:** A run can produce output even if verification warns, but a "passed" verification is required for `--publish` or `--share`.
- **Config cascade:** `~/.kasw/config.toml` → project `.kasw.toml` → env vars → CLI flags.

### 5.3 Error Handling Philosophy

- **Fail loud, fail fast:** Missing API key → immediate error with setup link.
- **Partial success:** If 2 of 10 subagents fail, report what succeeded and why the others failed.
- **Retry with backoff:** Provider 429/5xx retried up to 3 times with jitter.
- **Human escalation:** If cost estimate exceeds budget, pause and ask.

## 6. Go-to-Market & Community

### 6.1 Positioning

> "Kimi Agent Swarm Skill is the evidence-backed research engine for people who think in sources, not summaries."

Differentiators:
- Not a chatbot. Produces ledgers, verification reports, and audit trails.
- Honest about limits: no false claims of hosted Kimi Agent Swarm parity.
- Hackable provider system; bring your own search backend.
- Open-source; no black-box ranking.

**v0.9 — "Power User"**

## 7. Quality & Reliability

### 7.1 Testing Pyramid

- **Unit:** scorer, verifier, provider parsing (current: 16 tests → target: 100+).
- **Integration:** each provider with recorded fixtures.
- **E2E:** CLI from install to synthesis for each profile.
- **Benchmark:** golden-answer evaluation for 6 fixtures.

### 7.2 CI/CD Maturity

- Matrix tests across macOS/Linux/Windows.
- Nightly provider health checks (using minimal cost).
- Automated dependency updates with PRs.
- Release notes generated from conventional commits.
- Automated smoke test on published npm package.

### 7.3 Observability

- Structured logs (`--log-level info|debug`).
- Run-level telemetry (opt-in): duration, provider, depth, success rate.
- Public status page for built-in provider health.

## 8. Security & Compliance

- API keys only via env vars or keychain integration; never logged.
- No source code or user prompts sent to telemetry without opt-in.
- `.gitignore` templates for `.runs/`, `.private-references/`, `.env`.
- Provider data retention policy documented.
- Clear MIT license + THIRD_PARTY_NOTICES.

## 11. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Hosted Kimi Agent Swarm launches and makes this niche redundant | Medium | High | Double down on auditability, custom providers, and local-first trust. |
| Provider costs scare away users | High | High | Mock-first setup, dry-run, budgets, free-tier provider recommendations. |
| Model hallucinations undermine trust | Medium | High | Verifier-first design, source ledgers, confidence labels, human review prompts. |
| Community doesn't form | Medium | Medium | Office hours, benchmarks, contributor ladder, content marketing. |
| Scope creep turns this into a generic agent framework | Medium | Medium | Strict mission: evidence-backed wide search. Reject unrelated features. |
| API key leaks in user repos | Low | High | Docs + tooling to detect keys, never log keys, keychain integration. |
